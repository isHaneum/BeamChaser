# 커뮤니티 백필 계획

## 목적

친구/요청/팔로우/차단 구조를 다음 컬렉션 기준으로 정리한다.

- `users`
- `publicUsers`
- `friendRequests`
- `friendships`
- `follows`
- `blocks`

기존 `users/{uid}/friends`, `users/{uid}/blockedUsers`, 구형 `friendRequests.fromUserId/toUserId` 기반 데이터는 읽기 호환은 유지하되, 운영 데이터의 기준점은 위 컬렉션으로 전환한다.

## 원칙

- 첫 실행은 반드시 dry-run으로 진행한다.
- 실제 반영 전 Firestore Export를 먼저 만든다.
- `users`, `publicUsers`는 missing-field-only 방식으로 채운다.
- 이미 정상화된 최신 문서는 덮어쓰지 않는다.
- 사용자 이름은 기존 값을 우선 사용하고, 비어 있는 경우에만 안전한 기본값을 생성한다.
- 친구 관계의 기준은 `friendships.status == "active"`로 본다.
- 피드 표시 여부는 각 사용자의 `follows` 문서가 직접 소유한다.

## 사전 점검

1. 운영 프로젝트에서 Firestore Export를 수행한다.
2. 대상 프로젝트의 `users`, `publicUsers`, `friendRequests`, `friendships`, `follows`, `blocks` 문서 수를 기록한다.
3. `scripts/backfill-community-search.mjs`를 최신 코드 기준으로 고정한다.
4. 관리자 서비스 계정 권한과 대상 Firebase 프로젝트 ID를 다시 확인한다.
5. 사용자 이름 중복 후보를 미리 확인한다.

## 사용자 이름 전략

- 기존 `users.username`이 유효하면 그대로 사용한다.
- `publicUsers.username`만 있고 `users.username`이 비어 있으면 그 값을 승격한다.
- 둘 다 비어 있으면 표시 이름과 UID 일부를 이용해 기본 사용자 이름을 생성한다.
- 검색용 필드는 함께 채운다.
  - `usernameLower`
  - `searchId`
  - `displayNameLower`
- 충돌이 발생하면 숫자 suffix를 붙여 유일성을 확보한다.
- 이미 유효한 사용자 이름이 있는 문서는 재생성하지 않는다.

## 실행 순서

### 1. users / publicUsers 정규화

- `users`에서 누락된 필드만 채운다.
  - `username`
  - `totalDistance`
  - `runCount`
  - `photoURL`
- `publicUsers` 문서가 없으면 `users`를 기준으로 생성한다.
- `publicUsers` 문서가 있으면 검색/표시용 누락 필드만 채운다.

이 단계가 먼저 끝나야 검색, 친구 카드, 요청 화면, 추천 계산이 안정화된다.

### 2. legacy friend link 이관

- `users/{uid}/friends` 또는 동등한 레거시 친구 링크를 읽는다.
- 사용자 쌍마다 정렬된 키로 `friendships/{friendshipId}`를 생성한다.
- 상태는 `active`로 기록한다.
- 필요하면 호환용 `friendRequests` accepted 문서를 생성한다.
- 각 사용자 자신의 `follows` 문서만 생성한다.
  - `A -> B`
  - `B -> A`

운영 기준에서 친구 관계는 `friendships`, 피드 노출은 각 방향별 `follows`로 분리한다.

### 3. legacy block 이관

- `users/{uid}/blockedUsers`를 `blocks`로 승격한다.
- 차단 관계가 있으면 해당 사용자의 `follows`를 비활성화한다.
- 관련 `friendships`가 active면 removed로 내린다.
- pending `friendRequests`가 있으면 canceled 또는 rejected 상태로 정리한다.

### 4. 요청 데이터 정리

- 기존 `friendRequests.fromUserId/toUserId`는 유지하되, `senderId/receiverId`가 비어 있으면 채운다.
- `requestId`, `updatedAt`, `respondedAt` 누락 여부를 보정한다.
- 동일 사용자 쌍에 중복 pending 요청이 있으면 최신 문서 하나만 남기고 나머지는 canceled 처리한다.

### 5. follows 정리

- 문서 ID는 `followerId_followingId` 형태로 통일한다.
- `followerId == 문서 소유자` 규칙을 만족하지 않는 문서는 새 키로 재작성한다.
- 친구 active 상태인데 `follows`가 없는 경우, 각 사용자 자기 방향 문서만 생성한다.
- 차단/삭제 상태와 충돌하는 active `follows`는 비활성화한다.

## Dry Run 절차

1. `scripts/backfill-community-search.mjs`를 dry-run 기본값으로 실행한다.
2. 생성/수정 예정 문서 수를 컬렉션별로 확인한다.
3. 표본 사용자 10명 정도를 골라 예상 변경 사항을 수동 검토한다.
4. 중복 username, 중복 pending request, 비정상 follow key가 있으면 실제 적용 전에 먼저 정리한다.

## 실제 반영 절차

1. Firestore Export 백업 생성
2. dry-run 재실행으로 직전 결과 재확인
3. 적용 모드로 실행
4. 실행 로그를 파일로 보관
5. 완료 후 컬렉션별 문서 수 재집계

## 사후 검증

- `publicUsers`에서 username/searchId/displayNameLower 누락이 없는지 확인한다.
- 친구 샘플 계정으로 다음 동작을 검증한다.
  - 검색으로 친구 찾기
  - 요청 보내기
  - 요청 수락/거절
  - 친구 끊기
  - 피드 숨기기/다시 보기
  - 차단/차단 해제
- `friendships.status == "active"`인 쌍은 양방향 `follows`를 가져야 한다.
- `blocks`가 있는 쌍은 active `friendships`가 남아 있지 않아야 한다.
- 피드 숨김은 상대 문서가 아니라 내 `follows` 문서만 바뀌는지 확인한다.

## 롤백 기준

다음 조건이 나오면 즉시 롤백한다.

- username 충돌이 대량으로 발생한 경우
- active friendship 수가 예상보다 크게 줄어든 경우
- block 이관 후 친구/피드 데이터가 잘못 대량 비활성화된 경우
- `publicUsers` 검색 필드가 광범위하게 비어 있는 경우

롤백은 사전 생성한 Firestore Export를 기준으로 수행한다.

## 운영 메모

- 규칙 배포 전후로 `firestore.rules`와 `firestore.indexes.json`을 함께 관리한다.
- 현재 로컬 셸에 Firebase CLI가 없을 수 있으므로, 배포 단계는 `firebase` 또는 `npx firebase-tools` 사용 가능 여부를 먼저 확인한다.
- 앱 배포 전에는 최소 한 번 이상 스테이징 데이터셋에서 같은 절차를 반복한다.
